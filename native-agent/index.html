    Fabric Native Agent¶

    The Fabric Native Agent is a standalone CLI tool that runs as a long-lived daemon on your infrastructure. It connects to the Fabric mesh, discovers devices automatically, evaluates deterministic rules, and uses an LLM for open-ended reasoning — all configured through CLI flags and a YAML config file.

    Install it anywhere. Point it at your Fabric mesh. It handles the rest.

    Installation¶

    pip install ai-fabric-agent

    This gives you the ai-fabric-agent command.

    Quick Start¶

    # Minimal — connect to local Fabric mesh, use Bedrock Claude, discover all devices
ai-fabric-agent --model-provider bedrock --model-id us.anthropic.claude-sonnet-4-5-20250929-v1:0

# Full setup — rules, caching, filtered discovery, goal
ai-fabric-agent \
  --nats-url nats://your-server:4222 \
  --credentials /path/to/agent.creds.json \
  --model-provider bedrock \
  --model-id us.anthropic.claude-sonnet-4-5-20250929-v1:0 \
  --rules rules.yaml \
  --enable-caching \
  --goal "Keep the warehouse temperature between 18°C and 24°C"

    The agent starts, connects to the Fabric mesh, discovers devices, loads rules, and begins its event loop. It runs continuously until you stop it.

    CLI Reference¶

    ai-fabric-agent [OPTIONS]

    Connection¶

    Flag Env Variable Description Default
    --nats-url NATS_URL Fabric mesh URL nats://localhost:4222
    --credentials NATS_CREDENTIALS_FILE Path to .creds.json Auto-discovery
    --zone TENANT Fabric tenant/namespace default
    --tls-ca NATS_TLS_CA_FILE CA certificate for TLS Auto-discovery

    Model Selection¶

    Flag Env Variable Description Default
    --model-provider FABRIC_MODEL_PROVIDER bedrock or arm-registry required
    --model-id FABRIC_MODEL_ID Model identifier (provider-specific) required
    --model-region AWS_REGION AWS region (Bedrock only) us-east-1
    --local-model-checkpoint — Path to local model checkpoint (Arm registry) —
    --quantization — Quantization level: fp16, int8, int4 (Arm registry) fp16
    --fallback-provider — Fallback model provider if primary is unavailable —
    --fallback-model-id — Fallback model identifier —

    Discovery¶

    Flag Env Variable Description Default
    --discover-all — Discover all device types true
    --device-types — Comma-separated list of device types to discover All
    --refresh-interval — Seconds between registry syncs 30
    --no-watch — Disable real-time device join/leave events —

    Rules¶

    Flag Description Default
    --rules Path to rules YAML file — (no rules)
    --rules-dir Path to directory of rule YAML files —
    --dry-run-rules Validate rules without starting the agent —

    Caching¶

    Flag Env Variable Description Default
    --enable-caching — Enable Redis LLM response caching Disabled
    --redis-url REDIS_URL Redis connection URL redis://localhost:6379
    --cache-ttl — Cache entry TTL in seconds 3600
    --managed-redis — Let the agent start/stop its own Redis instance —

    Agent Behavior¶

    Flag Description Default
    --goal High-level goal for the agent to pursue — (event-driven only)
    --system-prompt Override the default system prompt Built-in
    --config Path to YAML config file (replaces all flags) —
    --log-level Logging verbosity: debug, info, warn, error info
    --log-file Write logs to file instead of stderr stderr

    Configuration File¶

    Every CLI flag can also be set in a YAML config file. Pass it with --config:

    ai-fabric-agent --config agent.yaml

    # agent.yaml

connection:
  nats_url: nats://your-server:4222
  credentials: /path/to/agent.creds.json
  zone: default

model:
  provider: bedrock
  model_id: us.anthropic.claude-sonnet-4-5-20250929-v1:0
  region: us-east-1
  fallback:
    provider: arm-registry
    model_id: arm/llama3-8b-chat-optimized

discovery:
  device_types:
    - cleaning_robot
    - temperature_sensor
    - security_camera
  refresh_interval: 30
  watch_events: true

rules: rules.yaml              # or inline rules (see below)

cache:
  enabled: true
  redis_url: redis://localhost:6379
  ttl: 3600
  managed: false

goal: "Keep the warehouse temperature between 18°C and 24°C"
log_level: info

    CLI flags override config file values. Environment variables sit in between — config file < env var < CLI flag.

    Inbuilt Device Discovery¶

    The agent subscribes to the Fabric device registry and maintains a live device table. When devices come online, go offline, or update their capabilities, the agent's view updates automatically. You don't call discover_devices — the agent already knows what's available.

    # Discover everything
ai-fabric-agent --model-provider bedrock --model-id us.anthropic.claude-sonnet-4-5-20250929-v1:0

# Only discover specific device types
ai-fabric-agent --device-types cleaning_robot,temperature_sensor \
  --model-provider bedrock --model-id us.anthropic.claude-sonnet-4-5-20250929-v1:0

# Faster refresh cycle for latency-sensitive environments
ai-fabric-agent --refresh-interval 5 \
  --model-provider bedrock --model-id us.anthropic.claude-sonnet-4-5-20250929-v1:0

    The agent uses discovered device capabilities to build its tool set dynamically. If a new device type appears on the mesh, the agent can use it immediately — no restart needed.

    Deterministic Event Rules¶

    Not every decision needs an LLM. For known cause-and-effect patterns, define rules that execute instantly and deterministically.

    Rules File¶

    # rules.yaml

rules:
  - name: high-temperature-response
    description: "Activate cooling when temperature exceeds threshold"
    trigger:
      event: threshold_alert
      device_type: temperature_sensor
      condition:
        temperature_c: { gte: 80 }
    action:
      invoke:
        device_type: cooling_system
        function: activate
        params:
          mode: emergency

  - name: motion-detected-lights
    description: "Turn on lights when motion is detected after hours"
    trigger:
      event: motion_detected
      device_type: security_camera
      condition:
        time_of_day: { after: "18:00", before: "06:00" }
    action:
      invoke:
        device_type: smart_light
        function: turn_on
        params:
          brightness: 100

  - name: battery-low-alert
    description: "Log alert when any robot battery drops below 15%"
    trigger:
      event: battery_low
      device_type: "*"          # match any device type
      condition:
        battery_pct: { lt: 15 }
    action:
      log:
        level: warn
        message: "Low battery on {device_id}: {battery_pct}%"

    Using Rules from the CLI¶

    # Load a single rules file
ai-fabric-agent --rules rules.yaml \
  --model-provider bedrock --model-id us.anthropic.claude-sonnet-4-5-20250929-v1:0

# Load all YAML files from a directory
ai-fabric-agent --rules-dir /etc/fabric/rules/ \
  --model-provider bedrock --model-id us.anthropic.claude-sonnet-4-5-20250929-v1:0

# Validate rules without starting the agent
ai-fabric-agent --rules rules.yaml --dry-run-rules

    Rules are evaluated before the LLM sees the event. If a rule matches, the action fires immediately. If no rule matches, the event is passed to the LLM for reasoning.

    This gives you:

    • Predictable behavior for known scenarios — no variance between runs
    • Low latency — rule evaluation is sub-millisecond
    • Zero token cost for deterministic actions
    • Auditability — rules are version-controlled YAML, not emergent model behavior

    Amazon Bedrock Model Access¶

    Use any model available on Amazon Bedrock. AWS credentials resolve from the standard chain (environment, IAM role, config file).

    # Claude on Bedrock
ai-fabric-agent \
  --model-provider bedrock \
  --model-id us.anthropic.claude-sonnet-4-5-20250929-v1:0 \
  --model-region us-east-1

# Llama on Bedrock
ai-fabric-agent \
  --model-provider bedrock \
  --model-id meta.llama3-1-70b-instruct-v1:0 \
  --model-region us-west-2

# Mistral on Bedrock
ai-fabric-agent \
  --model-provider bedrock \
  --model-id mistral.mistral-large-2407-v1:0 \
  --model-region eu-west-1

    Switch models without changing any agent logic, rules, or configuration. Tools, prompts, and device interactions work the same regardless of which model is behind the reasoning.

    Arm Model Registry¶

    For edge deployments where latency or connectivity makes cloud models impractical, select a model from the Arm model registry. These models are optimized for inference on Arm-based hardware — Cortex-A, Neoverse, or Ethos-U processors.

    # Run a local Arm-optimized model
ai-fabric-agent \
  --model-provider arm-registry \
  --model-id arm/llama3-8b-chat-optimized \
  --quantization int8

# Use a local checkpoint file
ai-fabric-agent \
  --model-provider arm-registry \
  --local-model-checkpoint /opt/models/llama3-8b-arm.bin \
  --quantization int4

    This is useful when:

    • Devices are in air-gapped environments with no cloud access
    • Round-trip latency to a cloud model is too high for real-time control
    • You want to keep data on-premises for compliance

    Model Fallback¶

    Configure a primary and fallback model. If the primary is unavailable (local model fails to load, Bedrock returns 5xx), the agent falls back automatically.

    # Primary: local Arm model. Fallback: Bedrock Claude.
ai-fabric-agent \
  --model-provider arm-registry \
  --model-id arm/llama3-8b-chat-optimized \
  --fallback-provider bedrock \
  --fallback-model-id us.anthropic.claude-sonnet-4-5-20250929-v1:0

    Redis Response Cache¶

    Enable caching to store LLM responses in Redis. Identical or semantically similar queries hit the cache instead of the model, reducing latency and cost.

    # Enable caching with an existing Redis instance
ai-fabric-agent \
  --enable-caching \
  --redis-url redis://your-redis:6379 \
  --model-provider bedrock --model-id us.anthropic.claude-sonnet-4-5-20250929-v1:0

# Let the agent manage its own Redis instance
ai-fabric-agent \
  --enable-caching \
  --managed-redis \
  --model-provider bedrock --model-id us.anthropic.claude-sonnet-4-5-20250929-v1:0

# Custom TTL (seconds)
ai-fabric-agent \
  --enable-caching \
  --cache-ttl 7200 \
  --model-provider bedrock --model-id us.anthropic.claude-sonnet-4-5-20250929-v1:0

    What Gets Cached¶

    • Device queries — "What's the temperature in zone A?" returns a cached response if the same query was answered recently
    • Planning responses — complex multi-step plans are cached to avoid regenerating them for identical situations
    • Tool call sequences — when the same reasoning leads to the same tool calls, skip the model entirely

    Cache Invalidation¶

    The cache automatically invalidates entries when:

    • The device landscape changes (new device, device goes offline)
    • A rule is added or modified
    • The TTL expires
    • You manually flush it via the agent API

    Goal Inference from Device Capabilities¶

    When you provide a --goal, the agent doesn't just list devices — it reasons about what's possible given the devices that are online.

    ai-fabric-agent \
  --goal "Keep the warehouse temperature between 18°C and 24°C" \
  --model-provider bedrock --model-id us.anthropic.claude-sonnet-4-5-20250929-v1:0

    The agent will:

    1. Check discovered devices for temperature-related capabilities (get_reading, threshold_alert)
    2. Check for actuation capabilities (activate on cooling systems, set_temperature on HVAC)
    3. Set up monitoring using available sensor events
    4. Create a control loop using available actuator functions
    5. Fall back to alerting a human if no actuator is available

    If you add a new HVAC device to the mesh later, the agent picks it up through discovery and incorporates it into the plan — no restart needed.

    Capability Matching¶

    The agent maps high-level goals to device capabilities using the function names, descriptions, and parameter schemas that devices expose through @rpc() decorators:

    Goal: "Keep the warehouse safe"
  ├── security_camera → watch_feed(), detect_anomaly()
  ├── door_lock → lock(), unlock(), get_status()
  ├── motion_sensor → motion_detected (event)
  └── alarm_system → trigger_alarm(), silence_alarm()

    If a required capability is missing (no alarm system on the mesh), the agent reports what it can and cannot do — rather than silently ignoring part of the goal.

    Examples¶

    Minimal Daemon¶

    ai-fabric-agent \
  --model-provider bedrock \
  --model-id us.anthropic.claude-sonnet-4-5-20250929-v1:0

    Connects to the local Fabric mesh, discovers all devices, runs in event-driven mode. No rules, no caching, no goal — the agent responds to events using LLM reasoning.

    Production Deployment¶

    ai-fabric-agent \
  --config /etc/fabric/agent.yaml \
  --log-file /var/log/fabric-agent.log \
  --log-level info

    # /etc/fabric/agent.yaml
connection:
  nats_url: nats://nats.internal:4222
  credentials: /etc/fabric/agent.creds.json
  zone: warehouse-east

model:
  provider: bedrock
  model_id: us.anthropic.claude-sonnet-4-5-20250929-v1:0
  region: us-east-1

discovery:
  device_types:
    - cleaning_robot
    - temperature_sensor
    - security_camera
    - cooling_system
    - smart_light
  refresh_interval: 15

rules: /etc/fabric/rules/

cache:
  enabled: true
  redis_url: redis://redis.internal:6379
  ttl: 1800

goal: "Monitor warehouse conditions and respond to anomalies"

    Edge Deployment (Air-Gapped)¶

    ai-fabric-agent \
  --nats-url nats://192.168.1.10:4222 \
  --model-provider arm-registry \
  --local-model-checkpoint /opt/models/llama3-8b-arm.bin \
  --quantization int8 \
  --rules /opt/fabric/rules.yaml \
  --goal "Monitor and control factory floor equipment"

    No cloud access needed. Model runs locally on the edge device. Rules handle known scenarios deterministically. The LLM handles the rest.

    Rules-Only Mode (No LLM)¶

    ai-fabric-agent \
  --rules rules.yaml \
  --model-provider none \
  --log-level debug

    Run purely deterministic — no model, no token cost. Events that don't match a rule are logged but not acted on.
