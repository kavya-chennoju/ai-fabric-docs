    Fabric Agent Integration¶

    Give your AI agent the ability to discover and control physical devices on a Fabric mesh using ai-fabric-tools.

    ai-fabric-tools is the agent-side SDK. It provides four tool functions that any agent framework can use — Strands, LangChain, or plain Python. The tools connect to NATS, query the device registry, and invoke device functions over JSON-RPC.

    Installation¶

    # Core (plain Python tools, no framework dependency)
pip install ai-fabric-tools

# With Strands adapter
pip install "ai-fabric-tools[strands]"

# With LangChain adapter
pip install "ai-fabric-tools[langchain]"
    Extra What it adds
    (none) Core tools + NATS client (nats-py, nkeys)
    strands strands-agents — Strands @tool wrappers
    langchain langchain-core — LangChain StructuredTool wrappers

    Connecting to the Mesh¶

    Before using any tools, call connect() to establish the NATS connection. The connection is a thread-safe singleton — call it once at startup.

    from ai_fabric_tools import connect

connect()

    Connection Resolution Order¶

    Each setting (NATS URL, credentials, TLS, tenant) resolves in this order:

    1. Explicit parameter passed to connect()
    2. Environment variable
    3. Auto-discovery from well-known paths

    connect() Parameters¶

    connect(
    nats_url="nats://your-server:4222",
    zone="default",
    credentials="/path/to/agent.creds.json",
    tls_config={"ca": "/path/to/ca-cert.pem"},
    request_timeout=30.0,
)
    Parameter Type Description Default
    nats_url str NATS broker URL From env or auto-discovery
    zone str Fabric tenant/namespace "default"
    credentials str \| dict Path to .creds.json or credentials dict From env or auto-discovery
    tls_config dict TLS configuration (CA cert, client cert/key) From env or auto-discovery
    request_timeout float Default timeout in seconds for device RPC calls 30.0

    Environment Variables¶

    Variable Description Example
    NATS_URL NATS broker URL nats://your-server:4222
    NATS_CREDENTIALS_FILE Path to credentials JSON file /home/agent/.fabric/agent.creds.json
    NATS_JWT JWT token (alternative to credentials file) eyJ0eXAiOiJKV1Qi...
    NATS_NKEY_SEED NKey seed (used with NATS_JWT) SUAE...
    NATS_TLS_CA_FILE CA certificate for TLS /etc/fabric/ca-cert.pem
    NATS_TLS_CERT_FILE Client certificate for mTLS /etc/fabric/client-cert.pem
    NATS_TLS_KEY_FILE Client key for mTLS /etc/fabric/client-key.pem
    TENANT Fabric zone/namespace warehouse-east

    Auto-Discovery¶

    If no explicit parameters or environment variables are set, connect() walks up from the current working directory looking for a security_infra/ directory containing credentials and TLS certificates. This means connect() with no arguments works when run from anywhere inside a Fabric project tree.

    Search paths:

    • ./security_infra/credentials/ — credential files (orchestrator.creds.json, devctl.creds.json)
    • ./security_infra/certs/ — CA and client certificates
    • ../security_infra/, ../../security_infra/, etc. — walks up the directory tree

    Connection Examples¶

    export NATS_URL=nats://your-server:4222
export NATS_CREDENTIALS_FILE=/home/agent/.fabric/agent.creds.json
export TENANT=default

    from ai_fabric_tools import connect
connect()  # picks up env vars automatically

    from ai_fabric_tools import connect
connect(
    nats_url="nats://your-server:4222",
    credentials="/home/agent/.fabric/agent.creds.json",
    zone="warehouse-east",
)

    # Run from anywhere inside your Fabric project
cd /home/user/my-fabric-project/agents/
python my_agent.py

    from ai_fabric_tools import connect
connect()  # finds security_infra/ automatically

    Disconnect¶

    from ai_fabric_tools import disconnect
disconnect()  # closes NATS connection and releases resources

    Tool Functions¶

    ai-fabric-tools provides four tool functions. These are plain Python functions — they work without any agent framework.

    from ai_fabric_tools import (
    discover_devices,
    invoke_device,
    invoke_device_with_fallback,
    get_device_status,
)

    discover_devices¶

    See what devices are on the mesh, what functions they expose, and what events they emit.

    devices = discover_devices()
robots = discover_devices(device_type="robot")
    Parameter Type Description Default
    device_type str \| None Filter by type (fuzzy match — "robot" matches "cleaning_robot") None (all)
    refresh bool Bypass cache and query registry directly False

    Returns a list of device dicts:

    [
    {
        "device_id": "robot-001",
        "device_type": "cleaning_robot",
        "location": "warehouse-B",
        "status": {"availability": "available"},
        "functions": [
            {
                "name": "start_cleaning",
                "description": "Start cleaning a specified zone.",
                "parameters": {
                    "type": "object",
                    "properties": {
                        "zone": {"type": "string"},
                        "priority": {"type": "string", "default": "normal"}
                    }
                }
            }
        ],
        "events": ["cleaning_complete", "battery_low"]
    }
]

    The function schemas come directly from the device driver's @rpc() decorated methods — their names, docstrings, and type hints. No manual schema authoring needed.

    invoke_device¶

    Call a function on a specific device.

    result = invoke_device(
    device_id="robot-001",
    function="start_cleaning",
    params={"zone": "zone-A"},
    llm_reasoning="Camera detected spill in zone-A",
)
    Parameter Type Description Default
    device_id str Target device ID required
    function str Function name to call required
    params dict \| None Function parameters (check discover_devices for schemas) None
    llm_reasoning str \| None Why the agent is making this call — logged for observability None

    Returns:

    # Success
{"success": True, "result": {"zone": "zone-A", "status": "cleaning_in_progress"}}

# Failure
{"success": False, "error": "Device robot-001 is offline"}

    The llm_reasoning field is never sent to the device. It's logged on the agent side so you can trace back why the model made a particular call.

    invoke_device_with_fallback¶

    Same as invoke_device, but tries a list of devices in order until one succeeds.

    result = invoke_device_with_fallback(
    device_ids=["robot-001", "robot-002", "robot-003"],
    function="start_cleaning",
    params={"zone": "zone-A"},
    llm_reasoning="Need any available robot for zone-A spill",
)
    Parameter Type Description Default
    device_ids list[str] Devices to try, in preference order required
    function str Function name to call required
    params dict \| None Function parameters None
    llm_reasoning str \| None Decision rationale None

    Returns:

    # Second device succeeded
{"success": True, "device_id": "robot-002", "result": {"status": "cleaning"}}

# All failed
{
    "success": False,
    "error": "All devices failed",
    "failed_devices": [
        {"device_id": "robot-001", "error": "offline"},
        {"device_id": "robot-002", "error": "timeout"},
        {"device_id": "robot-003", "error": "busy"}
    ]
}

    get_device_status¶

    Check a single device's current state.

    status = get_device_status(device_id="robot-001")
    Parameter Type Description Default
    device_id str Device to query required

    Returns:

    {
    "device_id": "robot-001",
    "device_type": "cleaning_robot",
    "location": "warehouse-B",
    "status": {"availability": "available"},
    "functions": ["start_cleaning", "stop_cleaning", "get_battery_level"]
}

    Using with Agent Frameworks¶

    The core tool functions are plain Python. To use them with an agent framework, import from the corresponding adapter — each adapter wraps the same four functions with the framework's tool decorator.

    Strands¶

    from ai_fabric_tools import connect
from ai_fabric_tools.adapters.strands import (
    discover_devices,
    invoke_device,
    invoke_device_with_fallback,
    get_device_status,
)
from strands import Agent

connect()

agent = Agent(
    tools=[discover_devices, invoke_device],
    system_prompt="You monitor and control IoT devices.",
)
agent("What devices are online?")

    LangChain¶

    from ai_fabric_tools import connect
from ai_fabric_tools.adapters.langchain import (
    discover_devices,
    invoke_device,
)
from langchain_anthropic import ChatAnthropic
from langgraph.prebuilt import create_react_agent

connect()

model = ChatAnthropic(model="claude-sonnet-4-5-20250929")
agent = create_react_agent(model, [discover_devices, invoke_device])
agent.invoke({"messages": [{"role": "user", "content": "List all devices"}]})

    Plain Python (No Framework)¶

    from ai_fabric_tools import connect, discover_devices, invoke_device

connect()

# Use directly — no agent needed
devices = discover_devices()
for d in devices:
    if d["device_type"] == "temperature_sensor":
        reading = invoke_device(d["device_id"], "get_reading")
        print(f"{d['device_id']}: {reading}")

    Writing a Custom Adapter¶

    If your framework isn't covered, writing an adapter is four lines per tool:

    from your_framework import tool_decorator
from ai_fabric_tools.tools import (
    discover_devices as _discover_devices,
    invoke_device as _invoke_device,
    invoke_device_with_fallback as _invoke_device_with_fallback,
    get_device_status as _get_device_status,
)

discover_devices = tool_decorator(_discover_devices)
invoke_device = tool_decorator(_invoke_device)
invoke_device_with_fallback = tool_decorator(_invoke_device_with_fallback)
get_device_status = tool_decorator(_get_device_status)

    The core functions carry their own names, docstrings, type hints, and parameter schemas. Most framework decorators pick these up automatically.

    Event Subscriptions¶

    The four tool functions are stateless request-reply operations. For real-time event streams from devices, use the connection object directly:

    import asyncio
from ai_fabric_tools import connect, get_connection

connect()
conn = get_connection()

async def watch_events():
    async for batch in conn.subscribe_events(batch_window=10.0):
        for event in batch:
            print(f"{event['device_id']}::{event['event_name']} → {event['params']}")

asyncio.run(watch_events())
    Parameter Type Description Default
    batch_window float Seconds to collect events before yielding a batch 12.0
    device_id str \| None Filter events from a specific device None (all)

    For subscribing to specific NATS subjects:

    conn = get_connection()

async def on_message(msg):
    print(f"Received: {msg.data.decode()}")

await conn.async_subscribe("fabric.default.robot-001.event.>", on_message)

    Access Control¶

    Credentials determine what your agent can see and do. A credential scoped to warehouse-east will:

    • discover_devices — only returns warehouse-east devices
    • invoke_device — calls to out-of-scope devices return an authorization error
    • get_device_status — same scoping

    This is enforced at the NATS layer, not in the SDK. The tools don't filter — the messaging infrastructure does.

    Architecture¶

    ┌─────────────────────────────────────────────────────┐
│  Your Agent (Strands, LangChain, plain Python)      │
│                                                     │
│  from ai_fabric_tools.adapters.strands import ...   │
│        │                                            │
│        ▼                                            │
│  ┌─────────────────────────────────────┐            │
│  │  Adapter Layer (4 lines per tool)   │            │
│  │  strands_tool(_discover_devices)    │            │
│  └──────────────┬──────────────────────┘            │
│                 │                                    │
│        ▼                                            │
│  ┌─────────────────────────────────────┐            │
│  │  Core Tool Functions (plain Python) │            │
│  │  discover_devices()                 │            │
│  │  invoke_device()                    │            │
│  │  invoke_device_with_fallback()      │            │
│  │  get_device_status()                │            │
│  └──────────────┬──────────────────────┘            │
│                 │                                    │
│        ▼                                            │
│  ┌─────────────────────────────────────┐            │
│  │  Connection (singleton, thread-safe)│            │
│  │  NATS client + sync-to-async bridge │            │
│  └──────────────┬──────────────────────┘            │
└─────────────────┼───────────────────────────────────┘
                  │ JSON-RPC over NATS
                  ▼
         ┌────────────────┐
         │   NATS Server   │
         │   + JWT Auth     │
         └───────┬─────────┘
                 │
    ┌────────────┼────────────┐
    ▼            ▼            ▼
┌────────┐ ┌────────┐ ┌────────┐
│robot   │ │camera  │ │sensor  │
│-001    │ │-002    │ │-003    │
└────────┘ └────────┘ └────────┘

    The adapter layer is intentionally thin — one function call per tool. All logic lives in the core functions. The connection manages a dedicated asyncio event loop on a daemon thread, bridging sync tool calls to the async NATS client.
